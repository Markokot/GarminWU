+# Архитектурный разбор GarminCoach AI
+
+## 1. Технологический стек
+- **Frontend:** React 18 + Vite + wouter + TanStack Query + shadcn/ui (Radix primitives)
+- **Backend:** Express 5 + express-session + memorystore
+- **AI:** DeepSeek через OpenAI-compatible SDK
+- **Интеграции:** Garmin Connect и Intervals.icu
+- **Хранение:** JSON-файлы в `.data/`
+- **Валидация и типы:** Zod + общий `shared/schema.ts`
+
+## 2. Архитектурная схема
+Приложение построено как монолит в одном репозитории:
+1. **Клиент** (`client/`) отдает SPA-интерфейс.
+2. **Сервер** (`server/`) отдает API и в production — статику фронта.
+3. **Shared слой** (`shared/schema.ts`) содержит доменные типы и Zod-схемы для фронта и бэка.
+
+## 3. Backend-архитектура
+### 3.1 Точка входа
+`server/index.ts`:
+- проверяет обязательные env-переменные,
+- поднимает Express и HTTP server,
+- подключает JSON/body middleware,
+- включает API-логирование,
+- регистрирует маршруты,
+- в dev подключает Vite middleware, в prod — статику,
+- запускает сервер на `PORT` (default 5000).
+
+### 3.2 Слой маршрутов
+`server/routes.ts` содержит почти всю бизнес-логику API:
+- auth (`/api/auth/*`),
+- профиль,
+- интеграции Garmin/Intervals,
+- workouts,
+- chat,
+- admin stats.
+
+Это **контроллер + часть сервисного слоя в одном файле**. Подход рабочий для small/medium проекта, но усложняет масштабирование.
+
+### 3.3 Слой данных
+`server/storage.ts`:
+- интерфейс `IStorage`,
+- реализация `FileStorage` на in-memory `Map` + сериализация в JSON файлы,
+- bcrypt-хеширование пароля,
+- CRUD для users/workouts/messages.
+
+Сильная сторона — простота. Ограничения — отсутствие транзакционности, слабая конкурентная модель записи и узкое место при росте данных.
+
+### 3.4 Интеграции
+- `server/garmin.ts`: управление сессиями Garmin, lazy reconnect, конвертер внутренней структуры тренировки в Garmin DTO.
+- `server/intervals.ts`: проверка аккаунта и отправка event в Intervals API.
+
+### 3.5 AI-подсистема
+`server/ai.ts`:
+- формирует системный промпт,
+- добавляет контекст пользователя и Garmin-активностей,
+- добавляет до 30 сообщений истории,
+- вызывает DeepSeek,
+- парсит workout JSON из markdown-блока.
+
+## 4. Frontend-архитектура
+### 4.1 Компоновка приложения
+`client/src/App.tsx`:
+- глобальные провайдеры: QueryClient, Theme, Tooltip, Auth,
+- условная развилка по авторизации,
+- shell с sidebar + header + роутинг через wouter.
+
+### 4.2 Управление состоянием
+- Серверные данные: TanStack Query.
+- Локальные UI-состояния: React hooks.
+- Auth: контекст `client/src/lib/auth.tsx` + запрос `/api/auth/me`.
+
+### 4.3 Страницы и UX
+- `auth-page.tsx`: вход/регистрация.
+- `dashboard-page.tsx`: summary по активностям и тренировкам.
+- `coach-page.tsx`: чат, превью тренировки, save/push.
+- `settings-page.tsx`: профиль + подключения Garmin/Intervals.
+- `admin-page.tsx`: агрегированная статистика.
+
+UI системный и единый благодаря shadcn/ui, много data-testid, адаптивность базово присутствует.
+
+## 5. Что сделано хорошо
+1. **Единый shared schema слой** снижает рассинхрон типов между клиентом и сервером.
+2. **Прагматичная интеграция с внешними API** (Garmin, Intervals) + попытки восстановить сессию.
+3. **Неплохой UX-флоу для core сценария**: запрос → AI → превью → сохранить/отправить.
+4. **Шифрование чувствительных токенов/паролей интеграций** (`AES-256-GCM`).
+5. **Встроенная админ-аналитика** на базе существующего хранилища.
+
+## 6. Архитектурные риски и технический долг
+1. **God-file в `server/routes.ts`** — растет когнитивная сложность.
+2. **FileStorage как основной persistence** — риск потерь/гонок записи, нет миграций/транзакций.
+3. **Сессии и учетные данные Garmin в памяти процесса** — при рестарте придется делать lazy connect; в multi-instance сценарии не масштабируется.
+4. **Несогласованность с Drizzle config**: есть `drizzle.config.ts`, но фактически БД не используется в runtime.
+5. **Типовая ошибка в AI-слое**: `tsc` падает, т.к. в `extractWorkoutJson` отсутствует `sentToIntervals`.
+6. **Ролевая модель админа захардкожена** (`username === "Andrey"`).
+
+## 7. Оценка (10-балльная шкала)
+- **Качество кода:** **7/10**
+  - Плюсы: читаемость, понятная структура, typed contracts.
+  - Минусы: большой routes-файл, местами широкие `any`, есть compile-time дефект.
+
+- **Архитектура:** **6.5/10**
+  - Плюсы: логичное разделение client/server/shared.
+  - Минусы: слабая масштабируемость storage/session слоя, смешение responsibility в API-слое.
+
+- **Интерфейс (UX/UI):** **8/10**
+  - Плюсы: цельный дизайн, понятные флоу, хороший набор статусов и состояний.
+  - Минусы: часть форм очень длинные (settings), нет явных onboarding-подсказок для сложных интеграций на уровне wizard.
+
+## 8. Приоритетный roadmap улучшений
+1. Разделить `routes.ts` на модули (`auth`, `workouts`, `chat`, `integrations`, `admin`) + сервисы.
+2. Исправить TS-ошибку в `server/ai.ts` и добавить проверку CI (`npm run check`).
+3. Перевести storage на Postgres (уже намечено через Drizzle) с миграциями.
+4. Вынести Garmin session management в отдельный сервис-слой с абстракцией кэша (Redis при масштабировании).
+5. Ввести нормальную RBAC-модель вместо хардкода админа.
+6. Добавить observability: structured logs + минимальные метрики (ошибки внешних API, latency, конверсия чат→workout→push).
